1 Introducción

1.1 Propósito

El propósito de este documento es describir de manera detallada los requerimientos funcionales y no funcionales de la aplicación EDI 301, cuyo objetivo es automatizar la asignación de alumnos a familias de empleados, mejorar la comunicación y optimizar la gestión de eventos del programa EDI 301 en la universidad.

1.2 Alcance

La aplicación permitirá gestionar familias, asignar alumnos internos, enviar notificaciones, administrar eventos, generar reportes y garantizar la seguridad y privacidad de la información mediante integración con SQL Server.

1.6 Resumen

Este documento especifica los requisitos funcionales y no funcionales para el desarrollo de una aplicación web y móvil llamada “APLICACIÓN EDI 301 PARA LA GESTIÓN DE FAMILIAS Y ALUMNOS DE LA UNIVERSIDAD LINDA VISTA”, que se desarrollará a partir del semestre 2025-B. La aplicación busca resolver problemas en la gestión manual del programa EDI 301, como la asignación de alumnos a familias de empleados, la organización de eventos, la comunicación mediante notificaciones y la generación de reportes administrativos.

2 Descripción general

2.1 Perspectiva del producto

La aplicación EDI 301 se desarrolla como una solución tecnológica para automatizar el proceso manual de asignación de alumnos a familias de empleados en la Universidad Linda Vista, integrándose con la base de datos institucional para garantizar información actualizada y confiable. El sistema permitirá gestionar familias y alumnos, organizar y registrar eventos, enviar notificaciones, generar reportes y controlar accesos según roles (capellán, padres/empleados y alumnos), con el objetivo de centralizar la información, optimizar la comunicación y mejorar la administración del programa EDI 301.

2.2 Funcionalidad del producto

La aplicación EDI 301 para la gestión de familias y alumnos está diseñada para llevar a cabo las siguientes funcionalidades principales:

  • Gestión de Familias y Alumnos: Crear y administrar familias, asignar alumnos internos, actualizar su estado (activo, en práctica, egresado) y restringir cambios sin autorización.
  • Control de Acceso y Roles: Garantizar que los empleados y sus cónyuges puedan gestionar su familia, mientras que los alumnos solo tienen acceso de visualización, asegurando la privacidad de la información.
  • Notificaciones y Comunicación: Enviar avisos a padres sobre eventos, permitir confirmación de asistencia, registrar ausencias anticipadas y dirigir mensajes a grupos específicos.
  • Gestión de Eventos: Crear y administrar eventos con detalles (fecha, lugar, requisitos), registrar la participación de familias y permitir la carga de evidencias fotográficas bajo reglas establecidas.
  • Privacidad y Seguridad de Información: Proteger datos personales, gestionar notificaciones de cumpleaños, restringir la publicación de imágenes a las autorizadas por los padres e integrar la información con la base de datos institucional.
  • Administración y Reportes: Generar reportes sobre asignación y estado de alumnos, asistencia a eventos y listas filtradas, además de gestionar accesos para nuevos empleados.

2.4 Restricciones

Hardware: (Se asume disponibilidad de dispositivos adecuados).

Permisos:

  • Solo usuarios registrados pueden acceder a la aplicación.
  • Los permisos de usuario están jerarquizados: Administrador, Empleado y Estudiante. Cada rol tendrá acceso únicamente a las funciones correspondientes a su nivel.
  • La creación, modificación o eliminación de datos críticos (como calificaciones de limpieza o asignación de estudiantes a familias) solo puede ser realizada por usuarios con permisos de Administrador.

Gestión de seguridad:

  • Toda la información almacenada debe ser cifrada en la base de datos.
  • Las contraseñas de los usuarios deben cumplir con políticas de complejidad: mínimo 8 caracteres, incluyendo mayúsculas, minúsculas, números y caracteres especiales.

Accesos restringidos:

  • Los estudiantes solo pueden visualizar su información personal y los anuncios dirigidos a ellos.
  • Los empleados pueden visualizar y modificar datos de sus familias asignadas, pero no tienen acceso a la información de otros empleados.
  • Los administradores tienen acceso completo a todas las funciones, pero cualquier operación sobre datos sensibles requiere confirmación doble o autenticación adicional.

2.5 Suposiciones y dependencias

Suposiciones Dependencias
  • Infraestructura tecnológica existente: Se asume que la universidad cuenta con la infraestructura tecnológica necesaria (servidores, red y equipos de cómputo) para soportar la operación del sistema EDI 301 y garantizar su disponibilidad.
  • Disponibilidad y validez de datos: Se asume que los datos proporcionados por el sistema institucional de la universidad (por ejemplo, matrículas, nombres, correos, etc.) son correctos, actualizados y están disponibles para su uso mediante los servicios o bases de datos institucionales.
  • Adopción y capacitación de usuarios: Se asume que los usuarios finales (capellanes, empleados y alumnos) recibirán la capacitación adecuada y adoptarán el uso del sistema de manera voluntaria y responsable.
  • Estabilidad de las reglas de negocio: Se asume que las reglas de negocio y los procesos institucionales relacionados con la asignación de alumnos, gestión de familias y control de asistencia no cambiarán significativamente durante el desarrollo del sistema.
  • Funcionalidad de los dispositivos de escaneo: Se asume que los dispositivos móviles o lectores de código QR utilizados para el registro de asistencia funcionarán correctamente y contarán con conectividad estable.
  • Unicidad de las credenciales: Se asume que cada usuario registrado en el sistema posee credenciales únicas, evitando duplicidad de cuentas y garantizando la autenticidad en el acceso.
  • Entrega y funcionamiento de los lectores de códigos de barras (si aplica): El correcto funcionamiento del sistema depende de la entrega oportuna y del adecuado desempeño de los lectores o dispositivos utilizados para el escaneo de códigos QR o de barras durante los registros de asistencia.
  • Integración con sistemas existentes (si aplica): La implementación del sistema EDI 301 depende de la correcta integración con las bases de datos y sistemas institucionales existentes, como los registros de alumnos y empleados de la universidad.
  • Aprobaciones y retroalimentación de las partes interesadas: El avance y validación de los módulos del sistema dependen de la disponibilidad, revisión y aprobación por parte de los capellanes, personal administrativo y demás partes interesadas.
  • Cumplimiento de normativas legales: El sistema depende del cumplimiento de las políticas institucionales de protección de datos personales y de las normativas legales aplicables al manejo de información académica y administrativa.
  • Disponibilidad del equipo de desarrollo: El progreso del proyecto depende de la disponibilidad continua del equipo de desarrollo y de los recursos técnicos necesarios para cumplir con los plazos establecidos.

2.6 Evolución previsible del sistema

El sistema EDI 301 está diseñado con una arquitectura modular que permitirá su crecimiento y adaptación a nuevas necesidades institucionales. Se prevé la siguiente evolución:

  1. Ampliación de funcionalidades: Incluir reportes más completos, estadísticas del programa y herramientas de seguimiento.
  2. Integración con otros sistemas institucionales: Conectar la app con bases de datos académicas, asistencia y comunicación de la universidad.
  3. Escalabilidad y seguridad: Garantizar que el sistema soporte más usuarios, migrar a infraestructura en la nube y reforzar la seguridad con autenticación avanzada.

4.1 Casos de uso

<

CU-01: Iniciar sesión

ActoresUsuario (alumno/empleado), Sistema
DescripciónEl usuario accede con correo y contraseña.
PrecondicionesUsuario registrado y activo.
Flujo principal
  1. El usuario ingresa correo y contraseña.
  2. El sistema valida credenciales.
  3. El sistema genera Session_token y establece sesión.
  4. El sistema muestra la pantalla principal.
Flujos alternativos
  • 2a. Credenciales inválidas → muestra error, permite reintentar.
  • 2b. Usuario inactivo.
PostcondicionesSesión iniciada.

CU-02: Cerrar sesión

ActoresUsuario (alumno/empleado), Sistema
DescripciónEl usuario finaliza su sesión activa.
PrecondicionesSesión iniciada.
Flujo principal
  1. El usuario solicita cerrar sesión.
  2. El sistema invalida Session_token.
  3. El sistema redirige a la pantalla de acceso.
PostcondicionesNo hay sesión activa.

CU-03: Registrar usuario

ActoresAdministrador, Sistema
DescripciónRegistrar un usuario autenticado con permiso.
PrecondicionesSesión iniciada.
Flujo principal
  1. El Administrador ingresa datos obligatorios (matrícula/num. empleado).
  2. El sistema valida unicidad (correo, matrícula/num. empleado).
  3. El sistema crea la cuenta y registra Created_at.
  4. El sistema notifica el éxito.
Flujos alternativos
  • 2a. Datos duplicados o inválidos → mensaje de error.
PostcondicionesUsuario creado.

CU-04: Crear familia

ActoresAdministrador, Sistema
DescripciónSe crea un registro de familia para asociar integrantes.
PrecondicionesAdministrador con sesión iniciada.
Flujo principal
  1. El Administrador captura datos básicos de la familia (num. empleado/apellidos).
  2. El sistema valida y crea la familia (Created_at).
  3. El sistema confirma la creación.
Flujos alternativos
  • 2a. Datos duplicados o inválidos → mensaje de error.
PostcondicionesFamilia registrada.

CU-05: Asignar integrante a familia

ActoresAdministrador, Sistema
DescripciónAsociar un usuario a una familia con su rol.
PrecondicionesFamilia y usuario existentes.
Flujo principal
  1. El Administrador busca usuario por matrícula/correo.
  2. Selecciona familia y rol.
  3. El sistema registra la relación.
  4. El sistema confirma.
Flujos alternativos
  • 1a. Usuario no existe → sugerencia de crear.
PostcondicionesIntegrante asociado.

CU-06: Consultar familia

ActoresUsuario (Alumno/Empleado/Administrador), Sistema
DescripciónVisualizar la composición de una familia.
PrecondicionesSesión iniciada.
Flujo principal
  1. El usuario ingresa criterio de búsqueda.
  2. El sistema lista coincidencias.
  3. Usuario selecciona familia.
  4. El sistema muestra los integrantes.
Flujos alternativos
  • 2a. No se encuentran coincidencias.
PostcondicionesFamilia visible.

CU-07: Publicar anuncio

ActoresAdministrador
DescripciónCrear una publicación informativa para la comunidad.
PrecondicionesSesión iniciada y permiso para publicar.
Flujo principal
  1. Redacta título y contenido.
  2. El sistema guarda la publicación (Created_at).
  3. El sistema notifica y envía avisos.
PostcondicionesPublicación visible.

CU-08: Compartir una publicación

ActoresUsuario (alumno/empleado/administrador), Sistema
DescripciónCrear una publicación para la comunidad.
PrecondicionesSesión iniciada y permiso.
Flujo principal
  1. El Usuario redacta contenido.
  2. El Empleado aprueba.
  3. El sistema guarda (Created_at).
  4. El sistema notifica.
PostcondicionesPublicación visible.

CU-09: Reaccionar a publicación

ActoresUsuario, Sistema
DescripciónRegistrar un “me gusta”.
PrecondicionesSesión iniciada; publicación existente.
Flujo principal
  1. Usuario toca “like”.
  2. El sistema alterna estado y actualiza conteo.
PostcondicionesReacción registrada.

CU-10: Comentar publicación

ActoresUsuario, Sistema
DescripciónEl usuario agrega un comentario.
PrecondicionesSesión iniciada; publicación existente.
Flujo principal
  1. El sistema valida longitud/formato.
  2. Guarda el comentario y lo muestra.
PostcondicionesComentario publicado.

CU-11: Crear evento

ActoresAdministrador, Sistema
DescripciónAlta de un evento institucional.
PrecondicionesSesión iniciada; permiso para eventos.
Flujo principal
  1. Captura datos del evento.
  2. El sistema valida y registra (Created_at).
  3. Notifica la creación.
PostcondicionesEvento registrado.

CU-12: Consultar eventos

ActoresUsuario, Sistema
DescripciónListar y ver detalle de eventos.
PrecondicionesSesión iniciada.
Flujo principal
  1. Usuario abre “Eventos”.
  2. El sistema lista próximos.
  3. Usuario abre detalle.
PostcondicionesDetalle visible.
